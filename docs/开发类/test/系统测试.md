
### 📝 GIMP AI Mentor 系统测试计划与测试程序 (积分版)

**文档版本：** V1.1
**测试对象：** GIMP AI Mentor Python-Fu 插件
**当前环境参考：** Windows 10/11, GIMP 2.10.34, Python 3.8.10

#### **1. 概述**

**1.1 系统需求概述**
本系统旨在开发一款基于Python-Fu的GIMP插件，核心目标是利用大语言模型（LLM）技术，将GIMP复杂的操作逻辑转化为自然语言交互，充当用户的“AI导师”。系统需实现以下核心价值：
*   **降低门槛：** 通过自然语言输入修图需求，自动生成操作指南。
*   **技能习得：** 通过可视化的步骤引导，帮助用户理解并掌握专业技法（“白盒化引导”）。
*   **非破坏性编辑：** 所有修改默认在临时图层进行，确保原始数据安全。
*   **工作流复用：** 支持将修图流程保存为本地“配方”（Recipe）进行固化和分享。

**1.2 系统设计与实现概述**
系统采用分层插件架构，作为GIMP的原生Python插件运行。
*   **架构模式：** 分层架构（表现层/业务逻辑层/AI交互层/GIMP集成层）。
*   **核心驱动：** 状态机模式（State Machine）管理引导流程。
*   **关键技术：** 多线程异步处理（避免阻塞主线程）、非破坏性编辑内核（调整图层/蒙版）、结构化JSON指令解析。
*   **数据流向：** GIMP宿主环境 -> 图像/状态读取 -> 云端AI API -> 结构化指令 -> 本地执行/预览 -> 用户确认/撤销。

**2.2 测试需求IADT分解**
基于SDD功能设计与接口设计，分解如下测试维度：
1.  **功能交互测试 (Interaction)：** 验证UI操作与GIMP宿主的联动。
2.  **AI数据流转测试 (Data Flow)：** 验证图像Base64编码、Prompt拼接、JSON解析的准确性。
3.  **指令执行测试 (Action)：** 验证JSON指令映射为GIMP PDB调用的正确性。
4.  **非破坏性测试 (Non-Destructive)：** 验证预览图层机制、撤销/重做历史栈的完整性。
5.  **异常与安全测试 (Exception)：** 网络超时降级、恶意配方文件校验、API密钥管理。

**2.3 测试实施阶段**
*   **阶段一（冒烟测试）：** 验证插件加载、面板显示、基础配置读写。
*   **阶段二（核心功能测试）：** 覆盖“意图输入 -> AI分析 -> 步骤引导 -> 预览执行”的全链路。
*   **阶段三（配方与管理测试）：** 验证配方的导入、导出、内置预设加载。
*   **阶段四（非功能测试）：** 性能指标验证、兼容性测试、安全性扫描。
*   **阶段五（回归测试）：** 修复缺陷后，针对相关模块进行回归验证。

**2.4 通过准则 (积分制)**
1.  **总分要求**：测试结束时，系统剩余积分必须 **≥ 65分**。
2.  **红线要求**：P0级核心用例（如非破坏性编辑、AI链路）若失败导致扣分超过15分，直接判定为不通过。
3.  **性能达标**：UI响应延迟 ≤ 300ms，全链路AI请求 ≤ 30秒，内存峰值 ≤ 500MB。
4.  **数据安全**：严格遵循非破坏性原则，原始图层在任何操作下均不可被直接修改。

#### **4. 测试实施与缺陷跟踪**

**1. 缺陷生命周期管理：**
*   **入库 (Submit)：** 测试人员提交缺陷，附带日志ID、截图、复现步骤。
*   **分配 (Assign)：** 测试组长审核后分配给开发人员。
*   **解决 (Resolve)：** 开发人员修复代码，提交版本库。
*   **验证 (Verify)：** 测试人员在新版本中验证缺陷是否关闭。
*   **归零 (Close)：** 验证通过，缺陷关闭；验证失败，重新打开。

**2. 回归测试策略：**
*   **每日构建 (Daily Build)：** 每次代码提交后，自动运行冒烟测试（F-01, F-03）。
*   **版本发布前：** 必须执行全量测试（功能+性能）。
*   **缺陷修复后：** 必须对修复的缺陷进行验证，并对其关联模块进行回归测试。
